# PnP Bridge 1.3.0

First-use setup gets dramatically faster: PnP Bridge now **pre-matches sites
and template variables automatically**, and template secrets (RADIUS/TACACS
keys, SNMP communities, AES keys, passwords) can be stored **encrypted** and
used as Day-N variables.

## New

- **Auto-suggest site mappings** (Settings → Site Mapping): one click
  pre-pairs unmapped NetBox sites with the Catalyst Center hierarchy using
  fuzzy token matching with abbreviation support (`FFM-DC1 →
  Global/Germany/Frankfurt/DC1`). Each suggestion shows a confidence badge;
  review, correct, then save — nothing is stored automatically.
- **Auto-suggest Day-N variable mappings** (Settings → Day-N Variables):
  pick a CCC template and PnP Bridge matches its variables against your
  actual NetBox data — device fields, custom fields, and config contexts
  discovered from a sample device — with a network-vocabulary synonym
  dictionary (`HOSTNAME → device.name`, `MGMT_IP →
  device.primary_ip4.address`, `SYSLOG_HOST →
  device.config_context.syslog.host`). Unmatched variables are added as
  empty rows and stay manual entry in the wizard.
- **Template secrets** (Settings → Day-N Variables): store named secrets
  encrypted (Fernet) and map them to template variables as
  `secret.<name>` — they are auto-suggested too (`RADIUS_KEY →
  secret.radius_key`). Values are **write-only**: masked in the UI, job
  records, logs, and every API response. The plaintext is decrypted
  just-in-time for the deploy call to Catalyst Center and exists nowhere
  else — covered by a dedicated test.

## Upgrade notes

- Migration `0007` (template secrets table) runs automatically at startup.
- Secrets are encrypted with the same key as credentials — keep backing up
  `/data` (DB + `secret.key`) together.
