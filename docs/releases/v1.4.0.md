# PnP Bridge 1.4.0

Day-N variables now mirror the real Catalyst Center IT-DayN templates, and you
can verify the resolved values against a real device before deploying.

## New

- **Catalyst-Center Day-N derivations from NetBox.** In addition to plain
  dot-paths, the resolver now computes the flat values the real IT-DayN
  templates use — matching the `nikor30/netbox_cc_dayn` mappings:
  - `device.uplink_ports` — cabled interface names, e.g. `Te1/1/3,Te1/1/4`
  - `device.uplink_switch` — the uplink's far-end device (from NetBox cabling)
  - `device.site_vlans` — all VLANs of the device's site as `(vid,name);(vid,name);…`
  - `device.support_contact` — site contact with role "Local IT", falling back
    to a device contact, then the tenant name
  Together with the existing paths (`device.site.name`, `device.location.name`,
  `device.rack.name`, `device.role.name`, `device.asset_tag`, …) this covers the
  full IT-DayN variable set (site_full_name, building_room, rack_id, device_role,
  asset_id, uplink_ports, uplink_switch, arrVLANs, support_contact). The
  auto-suggester proposes them from the real variable names.
- **Verify against a real device.** Settings → Day-N Variables has a new
  "Verify against a real device" panel: enter a device **serial number** and
  PnP Bridge resolves the current mappings against that real NetBox device,
  showing variable → source → resolved value. Read-only — nothing is changed,
  and secret-sourced variables stay masked. Use it to check the values against
  reality (e.g. a Catalyst Center template export) before deploying.

## Notes

- The `support_contact` derivation uses NetBox `contact-assignments`; the exact
  query params vary by NetBox version — use the serial preview to confirm the
  value on your instance. See CLAUDE.md §6.2.
